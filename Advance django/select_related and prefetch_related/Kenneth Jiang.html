    The ONE Thing that Rails scaffolding does wrong

    I had never been a big fan of any kind of code-gen until I came across Rails scaffolding 10 years back. I have to admit that RoR brought code-gen to a new level compared to all other similar mechanisms that came before it.

    However, a recent painful experience with RoR scaffolding reminded me of the reasons why I used to resent code-gen.

    What happened

    Last week I needed to build a RoR project from scratch, after not doing any serious RoR projects for 2 years. I had no idea about what's new in Rails 5 while barely knowing what was going into Rails 4. "No problem," I told myself, "I will just do rails new my_project and I will take off flying."

    It turned out that I was right. At the beginning.

    I worked on that fresh-new project for quite a few hours, adding a bunch of models, adding and configuring gems such as Devise. Long story short, significant amount of code changes went into the project. I was flying and it felt good. The same'ole good feeling of RoR...

    The smooth journey came to a squeaking stop when I was ready to add ActiveAdmin.

    As it turned out, ActiveAdmin doesn't have a robust support for Rails 5. 

    "Well," I scratched my head, "I really need ActiveAdmin but I don't think Rails 5 offers anything that I need but Rails 4 doesn't". So time to downgrade to Rails 4. And it should be as simple as changing Gemfile:

    -gem 'rails', '~> 5.0.1'
    +gem 'rails', '~> 4.2.6'

    WRONG!!!

    This is what happened:

    $ rails generate active_admin:install
    /app/config/environments/development.rb:53:in `block in <top (required)>': uninitialized constant ActiveSupport::EventedFileUpdateChecker (NameError)
    from /usr/local/bundle/gems/railties-4.2.7.1/lib/rails/railtie.rb:210:in `instance_eval'
    from /usr/local/bundle/gems/railties-4.2.7.1/lib/rails/railtie.rb:210:in `configure'
    ...

    Apparently the error was caused by a new setting introduced by Rails 5. That setting was added to the project by RoR scaffolding.

    The bigger problem is: even if I managed to get past this error by removing that particular setting, there are probably a bunch of other subtle or not-so-subtle differences between Rails 4 and Rails 5. And any of these differences may stop Rails from launching, or worse, cause some hidden problems that don't manifest themselves until much later. I did a quick diff between code generated by Rails 5 vs Rails 4, and more than 40% of the files have some differences that I don't fully understand.

    So I was left with 2 options:
    1. Starting a new project by scaffolding from a Rails 4 template, then applying all the changes I have made so far. Talk about waste of time!
    2. Charging forward with downgrading existing project to Rails 4, hoping to uncover and resolve all compatibility issues along the way. Talk about uncertainty on how long it will take.
    None of them is pretty.

    Code-gen is easily abused to become the attempt to hide skeleton in the closet. It won't be pretty if you have to poke deep into it.

    What could have been done differently in RoR scaffolding?

    RoR scaffolding, and for that matter, any code-gen implementations, can mitigate its short-coming by adhering to ONE principle: use code-gen only to remove repetitive work; not as a way to hide skeleton.

    Ideally no skeleton is created in a system. But as a sane programmer, I know it's close to impossible to have no skeletons in any software but the most trivial ones.

    So when you have to deal with skeletons, hide them with 'convention over configuration', not code-gen.

    There are actually some good examples in RoR. For instance, the model file generated by RoR scaffolding looks like this:

    class MyModel < ApplicationRecord
    end

    No skeletons. And as you can imagine, you will hardly run into issues of version incompatibility with code as succinct at this.

    Yet Another Network Simulator - Light, Fast, Docker-based

    There is no short of Network Simulators. For example, Brian Linkleter has done an impeccable job at maintaining a comprehensive list of all open source Network Simulators.

    However, when I tried to set up a simulated network to test some IPV6 configuration a few weeks back, I had such a difficult time to get one up and running. For instance, I was hoping CORE would be easy to set up and fast to run because it is, comparatively speaking, new kid on the block and uses LXC. However I struggled with it for a few hours, and it still steadfastly refused to deploy even the simplest simulated network on a pretty beefy AWS EC2 xlarge instance.

    Two commons problems that render most of these tools difficult to set up are:
    • They run on either Linux or Windows. But I have a Mac.
    • Most of them require GUI. This means that I have to set up VNC in AWS which is a pain in the *ss to begin with.
    That was when I asked "Why the hell are there no command line-based Network Simulators that are easy to install, configured, and run? On both Linux and Mac!"     And this is how YANS was born. It is based on Docker, but you don't have to know how Docker works to use YANS.

    One thing shocked me after I implemented it was how BLAZINGLY FAST it runs. In this screen recording below, it took mind-boggling 2.9 seconds to launch a network with 3 links and 6 nodes:

    Run Bluetooth service in Python on Intel Edison

    Intel Edison Arduino board has Bluetooth built in. However it took me quite a while to figure out how to do Bluetooth in Python.

    1. Unblock Bluetooth device

    When Edison is booted, the Bluetooth device is soft-blocked for some unknown reason. To unblock it:
    # rfkill unblock bluetooth
    # hciconfig hci0 up

    To verify that Bluetooth service is successfully unblocked:
    # rfkill list

    Verify that Bluetooth device is now working:
    # hcitool dev

    The output should have an "hci0" entry such as:
    Devices:
    hci0 58:A8:39:00:74:CB

    If everything works, add to /etc/rc.local (Create this file it it's missing) so that Bluetooth device will be unblocked every time device reboots:
    rfkill unblock bluetooth

    2. Enable Serial Port Profile

    Edit /etc/systemd/system/dbus-org.bluez.service and add -C after ExecStart=/usr/lib/bluez5/bluetooth/bluetoothd -E. 

    The reload bluetooth service and add Serial Port Profile:
    # systemctl stop bluetooth
    # systemctl start bluetooth
    # sdptool add SP 


    3 Install pybluez

    Follow the instruction at http://kennethjiang.blogspot.com/2016/11/install-pip-on-intel-edison-using-opkg.html

    Then
    # pip install pybluez


    4 Code away!

    You can find some code examples at https://github.com/karulis/pybluez/tree/master/examples/simple

    Install Pip on Intel Edison using OPKG


    This is tested as of November 2016 on the latest image of EDISON-3.0, which appears to be "Poky (Yocto Project Reference Distro) 1.7.2"

    1. Setup the Edison repo by replacing anything you have in the /etc/opkg/base-feeds.conf file with the following:

    src/gz all http://repo.opkg.net/edison/repo/all
    src/gz edison http://repo.opkg.net/edison/repo/edison
    src/gz core2-32 http://repo.opkg.net/edison/repo/core2-32

    2. Run the following command:

     opkg update

    3. Install pip by running the following commands:

    wget https://bootstrap.pypa.io/get-pip.py --no-check-certificate 
    python get-pip.py

    4. Install setuptools by running the following commands:

    wget --no-check-certificate https://bootstrap.pypa.io/ez_setup.py
    python ez_setup.py --insecure
    pip install --upgrade setuptools

    Per-Request SQL Caching in Django

    Per-Request SQL Caching in Django

    The problem

    In Django you have a much higher than you'd expect chance to run the same query multiple times in just 1 request. One of the many reasons why it is so is the "N+1 problem".

    The consequences are hardly pleasant: your web app is slow to respond to user's requests, and it hits the database much harder than it should.

    The solution

    One straightforward solution to this problem is to cache all queries on per-request basis, and serve them off of the cache when the same query is run again. As a matter of fact, this is the default behavior in Rails and it is called SQL caching.

    Django provides many different caching mechanisms but it's a bit of mystery to me why per-request SQL caching was not one of them. I searched for PyPi packages and found none of them doing exactly that.

    So I had to implement it for myself. Thanks to an awesome blog by Chase Seibert, I was able to create django-sql-caching that you can easily plugged into your Django project and watch your web app being 20-80% faster (depending on how much repeated queries you originally had).

    django-sql-caching is currently being used in a fairly complex Django project (150k+ lines of code) at Rhumbix without any issues.

    The caveat

    I do want to point out a caveat for using django-sql-caching. Because django-sql-caching doesn't invalidates the cache when an object is updated in the middle of a request, subsequent identical queries will return outdated objects from the cache. For example, suppose this code in a Django view:

     user = User.objects.filter(first_name="Kenneth")[0]
    user.email = 'nosuch@email.com'; user.save()
    user = User.objects.filter(first_name="Kenneth")[0] # Broken. user.email will now be the old value

    This problem is less likely than you thought to happen. The 2nd query (the one after update) should be exactly the same as the 1st one to trigger this problem. For example the following code will work:

    user = User.objects.filter(first_name="Kenneth")[0]
    user.email = 'nosuch@email.com'; user.save()
    user = User.objects.filter(last_name="Jiang")[0] # This works!

    Or

    user = User.objects.filter(first_name="Kenneth")[0]
    user.email = 'nosuch@email.com'; user.save()
    user = user.refresh_from_db() # This too!

    Or

    user = User.objects.filter(first_name="Kenneth")[0]
    user.email = 'nosuch@email.com'; user.save()
    import threading; threading.local().query_cache = {}
    user = User.objects.filter(first_name="Kenneth")[0] # And this!

    Happy coding!

    Put S3 policies under version control in Django projects

    If your Django project uses S3 as media storage, and you need some access control that you have to use S3 policy, you may have realized that AWS S3 policy is a real pain to work with.

    S3 policy has a very cryptic syntax. Combined with not having a way to pre-fly a change, we have had the following problems around managing S3 policies:

    • Changing S3 policy becomes one of the riskiest thing we have to do in production. Even if we have a staging environment which is kept as close to production as possible, we can't directly copy S3 policies from staging to production because all policies contains information that is specific to environment, such as bucket names and user names.
    • Changing S3 policy becomes an ad-hoc process. It involves little more than one of the engineers with AWS access logging to AWS console, and changing the S3 policy to be something that hopefully will work.
    • If things are broken related to S3 permission, it's not clear to us if it was because we made a chance to S3 policy, or it was because we made a code change.
    I decided to take this into my own hands. Since I'm a software engineer, the solution is easy: put s3 policies under version control!

    If you have an Django project, I have made it easy for you. Check out django-s3-policy-sync. The only thing you need to do is to put your S3 polices in files, and put these files in the same git repo as your Django project. For more details, check out README file in django-s3-policy-sync.

    jQuery in Django admin site

    Django admin site by default includes jQuery. And since you can override Django admin forms with and custom form, you can customize a Django admin page of custom forms with jQuery capability such as AJAX. Posts like this give details on how you can do this.

    The caveat: the way Django admin site includes jQuery is very tricky. For one reason or another Django admin decided to include 2 versions of jQuery. As shown in the screenshot, one jQuery version is loaded from Django admin package, another version is http://ajax.googleapis.com/ajax/libs/jquery/2.1.4/jquery.min.js


    As the result, if you wrote jQuery the same way you would on a regular web page, it would give you the error message shown on the screenshot:
    Uncaught TypeError: $ is not a function

    The solution

    Your javascript should be initialized in the way shown below:

    (function($) {
        // < start of closure
        // within this block, $ = django.jQuery
        $(document).ready(function() {

      // Your JavaScript
    
    });
})(django.jQuery);
    

    

    

    Sometimes even this will not work. If you still get the same error even if your JavaScript is initialized as above, you can do an uglier but equivalent version of the same code:
    

    

    

    (function(jq) {
    

        // < start of closure
    

        // within this block, jq = django.jQuery
    

        $(document).ready(function() {

      // Your JavaScript. But you can't use $ as you would with 
    

          // jQuery-based JavaScript. Instead, you should replace $ with jq
    

    
    });
})(django.jQuery);
    

    

    

    
It's not the pretties thing you can ask for. But hey you get the access to jQuery to do things like AJAX calls!

Advanced usage of prefetch_related and select_related in Django
  
  





    

    
  
    

    

    
One of the most frequent performance issues your Django app will run into is “N+1 queries”. The tools that Django provides to solve this performance issue are prefetch_related and select_related.
    

    

    

    
I will not cover the basics on “N+1 queries” and prefetch_related/select_related in this blog. Please check the following links for really good introductions on them:
    

    

    

      

    • Performance: N+1 Query Problem
      • 

    • Django's select_related and prefetch_related
      • 

    

    

    
Instead I will focus on more “advanced” problems — the problems you will run into when your app becomes reasonably complicated.
    

    
How to identify N+1 queries
    

    
The best way to identify N+1 queries is to leverage a performance monitoring tool like New Relic while your app is running in production environment taking real traffic. Go to “Monitoring -> Transactions”. Click on one of the transactions, you will see a breakdown table similar to this:
    

    

    

    

    

    
Highlighted in the screenshot above is a likely culprit of N+1 query, as the same query was executed on average 9 times for each transaction (page view).
    

    

    

    
The second-to-the-best way to identify N+1 is to use django-debug-toolbar in your dev environment. Django-debug-toolbar presents the stats in a different ways than New Relic but the concept is exactly the same.
    

    
An example
    

    
Now you know what N+1 queries are, and how you can identify them, let’s use an example to show how we can apply these concepts in a situation more realistic than most other tutorials you can find.
    

    
Suppose your Django app has data model as follows:
    

    

    

    class User(models.Model):
    name = models.CharField(max_length=200)

class Post(models.Model):
    author = models.ForeignKey(User)

class Comment(models.Model):
    post = models.ForeignKey(Post)
    commenter = models.ForeignKey(User)
    hidden = models.BooleanField(default=False)
    

    

    

    
Case #1
    

    
Suppose, on /comments/[id] page, you need to display all the posts that the commenter has published. A naive way to write the code is:
    

    

    

    {% for post in comment.commenter.post_set.all() %}
    ...
    

    

    

    
Obviously you will run into N+1 problem here. Since “commenter” is a foreigner key from Comment to User, it sounds like you should use “select_related”:
    

    

    

    {% for post in comment.commenter.post_set.all().select_related('commenter') %}
    

    

    
If you run it, you will find that N+1 query is still there. Why? This is because select_related(‘commenter’) only preloads the related records from “User” table, but not all the related records from “Post” table. Here is the revised code:
    

    

    

    {% for post in comment.commenter.post_set.all().prefetch_related('commenter__post_set') %}
    

    

    
Learning #1: if you have related entities nested to another related entities, you need to use “prefetch_related” to go all the way to the very end of relationship chain to make sure every links in the middle are loaded too. And this is true even if some of the links consist of a “many-to-1” relationship with their owning entities.
    

    
Case #2
    

    
Suppose you want to display all the posts, and the comments that are not hidden, along with the commenter’s name. Again if you don’t consider N+1 problem, the code would be something similar to this:
    

    

    

    {% for post in Post.objects.all() %}
    {% for comment in post.comment_set.filter(hidden=False).all() %}
        {{ comment.commenter.name }}
        ...
    

    

    
Your first attempt to solve it may be:
    

    

    

    {% for post in Post.objects.all().prefetch_related('comment_set' %}
    {% for comment in post.comment_set.filter(hidden=False).all().select_related('commenter' %}
        {{ comment.commenter.name }}
        ...
    

    

    
If you measure it with New Relic or django-debug-toolbar you will find the the number of queries will be roughly cut in half. But it’ll be obvious that it’s still suffering from N+1 problem. Why?
    

    

    

    
The reason is, in the nested “for” loop, post.comment_set.filter(hidden=False) will always result in 1 query for each post, even if all comments for that post have been pre-loaded. Django can’t do query for the objects in memory, which makes sense.
    

    

    

    
The solution: preload everything in the first query, then do the filtering in code (as opposed to using Django’s filter). So the final code should look like:
    

    

    

    {% for post in Post.objects.all().prefetch_related('comment_set__commenter' %}
    {% for comment in post.comment_set.all() if not comment.hidden %}
        {{ comment.commenter.name }}
        ...

Configure XCode to build for different environments
  
  





    

    
  
    
The problem
    

    
At Rhumbix we have several environments for our RESTful API backend: Development, Test, and Production. This is pretty much a standard practice.
    

    

    

    
Similarly when we build our iOS app that calls these RESTful APIs, we want to have a switch in our build process so that the resulted ipa will connect to the correct environment. For example  Production build should connect to https://api.rhumbix.com, and Test build to http://api.test.rhumbix.com, etc.
    

    

    

    
There are other situations where we want to fine tune some variables based on which environment we are building the ipa for. For instance, the Testing build will point to a different Sentry DSN than the Production build.
    

    

    

    
The options
    

    
The 1st thought we had was to create different targets for our projects. But it turned out to be tricky for a number of reasons.
    

    

    

    
Some online posts suggested creating one scheme for each environment. That didn’t work out for us either.
    

    

    

    
Eventually we settled on using a lesser-known feature in Xcode called “Build Configuration”.
    

    

    

    
The details
    

    
1. Create Build Configurations
    

    

    

      

    • Go to “Your project” -> “Info” -> “Configuration”. 
      • 

    • Click on “+” and select “Duplicate Release Configuration” (or “Duplicate Debug Configuration” if you want those debugging build settings such as “debug symbols” or “build for active architecture”).
      • 

    

    

    

    

    

    

    
2. Add a file called “Environment.plist” in “Supported files” and make it look like this:
    

    

    

    

    

    
3. Create Environment.h and Environment.m:
    

    
// Envioronment.h
    

    
#import <Foundation/Foundation.h>
    

    
#import "DOSingleton.h"
    

    

    

    
@interface Environment : DOSingleton
    

    

    

    
@property (strong, nonatomic) NSString *apiBaseUrl;
    

    
@property (strong, nonatomic) NSString *sentryDSN;
    

    

    

    
@end
    

    

    

    

    

    
// Environment.m
    

    
#import "Environment.h"
    

    

    

    
@implementation Environment
    

    

    

    
- (id)init
    

    
{
    

    
    if (!self.isInitialized) {
    

    
        self = [super init];
    

    
        
    

    
        if (self) {
    

    
            NSString* configuration = [[[NSBundle mainBundle] infoDictionary] objectForKey:@"Configuration"];
    

    
            NSBundle* bundle = [NSBundle mainBundle];
    

    
            NSString* envsPListPath = [bundle pathForResource:@
    

    
                                       "Environments" ofType:@"plist"];
    

    
            NSDictionary* environments = [[NSDictionary alloc] initWithContentsOfFile:envsPListPath];
    

    
            NSDictionary* environment = [environments objectForKey:configuration];
    

    
            
    

    
            self.apiBaseUrl = [environment valueForKey:@"apiBaseUrl"];
    

    
            self.sentryDSN = [environment valueForKey:@"sentryDSN"];
    

    
        }
    

    
    }
    

    
    
    

    
    return self;
    

    
}
    

    

    

    
@end
    

    

    

    
4. Now our programmer can use this simple statement to get an environment-dependent variables simply like this:
    

    
[Environment sharedInstance].apiBaseUrl
    

    

    

    

    

    
Caveats around cocoapods:
    

    
Most likely you are using cocoapods as we do. If so you may have to do more.
    

    

    

    
1. If you have this error when you run `pod install`:
    

    
[!] CocoaPods did not set the base configuration of your project because your project already has a custom config set. In order for CocoaPods integration to work at all, please either set the base configurations of the target `RMBX` to `Pods/Target Support Files/Pods/Pods.debug.xcconfig` or include the `Pods/Target Support Files/Pods/Pods.debug.xcconfig` in your build configuration.
    

    

    

    
Make sure you wrap all your pods in a “target” statement in Podfile:
    

    
target 'RMBX' do
    

    
   pod 'AFNetworking', '~> 2.5'
    

    
   pod ...
    

    
   pod 'Crashlytics', '~> 3.0.11'
    

    
end
    

    

    

    
2. You may find it a problem that build settings in the corresponding configuration not being propagated to Pods your project depends on.
    

    
For instance, the problem we ran into was that the “DEBUG” preprorcessor macro was not set for Pods even when we duplicated our Test configuration from “Debug”.
    

    

    

    
The way to solve it is to add the following to your Podfile:
    

    
...
    

    
xcodeproj 'RMBX', {
    

    
  'Debug' => :debug,
    

    
  'Development' => :debug,
    

    
  'Test' => :debug,
    

    
  'Release' => :release,
    

    
}
    

    
...
    

    

    

    

    

    
Notes: We wouldn't have figured this out without the posts from these awesome people:

Simplicity series - writing stupid code
  
  





    

    
  
    

    
We all know the KISS principle - Keep It Simple and Stupid. However, even for the few developers who really talk about KISS, the emphasis has always been on "Simple", rather than on "Stupid".
    

    
After all, it sounds a lot better to say "I write simple code" than "I write stupid code".
    

    
My biggest problem with talking about only "Simple" but not "Stupid" is the implication that it's okay to write "simple but clever" code.
    

    
What do I mean? Why do I have problem with writing clever code?
    

    

    

    
Let's use a web application that I was involved in as an example. The application needed to model the relationship between workers and work shifts. Since this was obviously many-to-many relationship between worker and work shift, a no-brainer, stupid data model would be like the E-R diagram on the left.
    

    
If things were that simple and stupid in the real world!
    

    
At the beginning of the project, the business wisely pointed out that, although there was no plan to support this scenario any time soon, it was possible that we would allow 1 worker to switch between companies.
    

    
This suggestion obviously excited the engineers - now the data model wouldn't appear to be that stupid! Therefore the clever, future-proof version of the data model below:
    

    

    

    

    

    
"What wrong with that?" You may ask. "The 2nd data model looks perfectly fine to me!"
    

    
No there was nothing wrong with the 2nd data model itself. What was wrong was that it wasn't the "stupidest" data model at the time it was designed and implemented. So what happened afterward:
    

    

      

    • 6 months since the product was built, the business hasn't run into the "worker with 1+ companies" scenarios. Instead, the most frequent queries included "find work shifts for a worker_id" and "find workers for a work_shift_id".  If the team had gone with the 1st data model, these queries would have been extremely straightforward. But now in 2nd data model, there were all 3-table-join queries. This not only makes coding more complicated and error-prone, but also make ad-hoc queries and trouble-shooting harder and more time-consuming.
      • 

    • Now 6 months later, the business actually came to the engineering team and said "there is a change of plan - we want to give the worker option to start work shift without being enrolled in any company first". Now what? The engineering team scratched their heads. Let's add a "dummy company" to company table to indicate the workers are not enrolled in any company.  Fine. A lot of ugly if (company == dummy_company) ... else ...  were bound to be inserted to the code base, which certainly looks far less clever than it was originally designed.
      • 

    

    

    
 Cleverness can, and will most certainly, backfire. So don't forget the "Stupid" in KISS principle.
